<p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a> and <a href="/wiki/Information_theory" title="Information theory">information theory</a>, <b>data compression</b> or <b>source coding</b> is the process of <a href="/wiki/Encoding" title="Encoding" class="mw-redirect">encoding</a> information using fewer <a href="/wiki/Bit" title="Bit">bits</a> (or other information-bearing units) than an <a href="/wiki/Code" title="Code">unencoded</a> representation would use, through use of specific <a href="/wiki/Encoding" title="Encoding" class="mw-redirect">encoding</a> schemes.</p>
<p>Another concept related to compression is that of <a href="/wiki/Data_deduplication" title="Data deduplication">Data deduplication</a>. In computing, data deduplication is a specialized data compression technique for eliminating coarse-grained redundant data, typically to improve storage utilization.</p>
<p>As with any communication, compressed data communication only works when both the <a href="/wiki/Sender" title="Sender">sender</a> and receiver of the <a href="/wiki/Information" title="Information">information</a> understand the encoding scheme. For example, this text makes sense only if the receiver understands that it is intended to be interpreted as characters representing the English language. Similarly, compressed data can only be understood if the decoding method is known by the receiver.</p>
<p>Compression is useful because it helps reduce the consumption of expensive resources, such as <a href="/wiki/Hard_disk" title="Hard disk" class="mw-redirect">hard disk</a> space or transmission <a href="/wiki/Bandwidth_(computing)" title="Bandwidth (computing)">bandwidth</a>. On the downside, compressed data must be decompressed to be used, and this extra processing may be detrimental to some applications. For instance, a compression scheme for video may require expensive hardware for the video to be decompressed fast enough to be viewed as it is being decompressed (the option of decompressing the video in full before watching it may be inconvenient, and requires storage space for the decompressed video). The design of data compression schemes therefore involves trade-offs among various factors, including the degree of compression, the amount of distortion introduced (if using a <a href="/wiki/Lossy_data_compression" title="Lossy data compression" class="mw-redirect">lossy compression scheme</a>), and the computational resources required to compress and uncompress the data.</p>

<h2><span class="editsection">[<a href="/w/index.php?title=Data_compression&amp;action=edit&amp;section=1" title="Edit section: Lossless versus lossy compression">edit</a>]</span> <span class="mw-headline" id="Lossless_versus_lossy_compression">Lossless versus lossy compression</span></h2>
<div class="rellink relarticle mainarticle">Main articles: <a href="/wiki/Lossless_data_compression" title="Lossless data compression">Lossless data compression</a> and <a href="/wiki/Lossy_data_compression" title="Lossy data compression" class="mw-redirect">lossy data compression</a></div>
<p><a href="/wiki/Lossless_data_compression" title="Lossless data compression">Lossless</a> compression algorithms usually exploit statistical redundancy in such a way as to represent the sender's data more concisely without error. Lossless compression is possible because most real-world data has statistical redundancy. For example, in English text, the letter 'e' is much more common than the letter 'z', and the probability that the letter 'q' will be followed by the letter 'z' is very small. Another kind of compression, called <i><a href="/wiki/Lossy_data_compression" title="Lossy data compression" class="mw-redirect">lossy data compression</a></i> or <a href="/wiki/Perceptual_coding" title="Perceptual coding" class="mw-redirect">perceptual coding</a>, is possible if some loss of <a href="/wiki/Fidelity" title="Fidelity">fidelity</a> is acceptable. Generally, a lossy data compression will be guided by research on how people perceive the data in question. For example, the human eye is more sensitive to subtle variations in <a href="/wiki/Luminance" title="Luminance">luminance</a> than it is to variations in color. <a href="/wiki/JPEG" title="JPEG">JPEG</a> image compression works in part by "rounding off" some of this less-important information. Lossy data compression provides a way to obtain the best fidelity for a given amount of compression. In some cases, <i>transparent</i> (unnoticeable) compression is desired; in other cases, fidelity is sacrificed to reduce the amount of data as much as possible.</p>
<p>Lossless compression schemes are reversible so that the original data can be reconstructed, while lossy schemes accept some loss of data in order to achieve higher compression.</p>
<p>However, lossless data compression algorithms will always fail to compress some files; indeed, any compression algorithm will necessarily fail to compress any data containing no discernible patterns. Attempts to compress data that has been compressed already will therefore usually result in an expansion, as will attempts to compress all but the most trivially <a href="/wiki/Encryption" title="Encryption">encrypted</a> data.</p>
<p>In practice, lossy data compression will also come to a point where compressing again does not work, although an extremely lossy algorithm, like for example always removing the last byte of a file, will always compress a file up to the point where it is empty.</p>
<p>An example of lossless vs. lossy compression is the following string:</p>
<dl>
<dd>25.888888888</dd>
</dl>
<p>This string can be compressed as:</p>
<dl>
<dd>25.[9]8</dd>
</dl>
<p>Interpreted as, "twenty five point 9 eights", the original string is perfectly recreated, just written in a smaller form. In a lossy system, using</p>
<dl>
<dd>26</dd>
</dl>
<p>instead, the exact original data is lost, at the benefit of a smaller file.</p>
<h2><span class="editsection">[<a href="/w/index.php?title=Data_compression&amp;action=edit&amp;section=2" title="Edit section: Example algorithms and applications">edit</a>]</span> <span class="mw-headline" id="Example_algorithms_and_applications">Example algorithms and applications</span></h2>
<p>The above is a very simple example of <a href="/wiki/Run-length_encoding" title="Run-length encoding">run-length encoding</a>, wherein large runs of consecutive identical data values are replaced by a simple code with the data value and length of the run. This is an example of lossless data compression. It is often used to optimize disk space on office computers, or better use the connection <a href="/wiki/Bandwidth_compression" title="Bandwidth compression">bandwidth</a> in a <a href="/wiki/Computer_network" title="Computer network">computer network</a>. For symbolic data such as spreadsheets, text, <a href="/wiki/Executable_compression" title="Executable compression">executable programs</a>, etc., losslessness is essential because changing even a single bit cannot be tolerated (except in some limited cases).</p>
<p>For visual and audio data, some loss of quality can be tolerated without losing the essential nature of the data. By taking advantage of the limitations of the human sensory system, a great deal of space can be saved while producing an output which is nearly indistinguishable from the original. These lossy data compression methods typically offer a three-way tradeoff between compression speed, compressed data size and quality loss.</p>
<h3><span class="editsection">[<a href="/w/index.php?title=Data_compression&amp;action=edit&amp;section=3" title="Edit section: Lossy">edit</a>]</span> <span class="mw-headline" id="Lossy">Lossy</span></h3>
<p>Lossy <a href="/wiki/Image_compression" title="Image compression">image compression</a> is used in <a href="/wiki/Digital_camera" title="Digital camera">digital cameras</a>, to increase storage capacities with minimal degradation of picture quality. Similarly, <a href="/wiki/DVD" title="DVD">DVDs</a> use the lossy <a href="/wiki/MPEG-2" title="MPEG-2">MPEG-2</a> <a href="/wiki/Video_codec" title="Video codec">Video codec</a> for <a href="/wiki/Video_compression" title="Video compression">video compression</a>.</p>
<p>In lossy <a href="/wiki/Audio_compression_(data)" title="Audio compression (data)">audio compression</a>, methods of <a href="/wiki/Psychoacoustics" title="Psychoacoustics">psychoacoustics</a> are used to remove non-audible (or less audible) components of the <a href="/wiki/Audio_signal_processing" title="Audio signal processing">signal</a>. Compression of human speech is often performed with even more specialized techniques, so that "<a href="/wiki/Speech_encoding" title="Speech encoding" class="mw-redirect">speech compression</a>" or "voice coding" is sometimes distinguished as a separate discipline from "audio compression". Different audio and speech compression standards are listed under <a href="/wiki/Audio_codec" title="Audio codec">audio codecs</a>. Voice compression is used in <a href="/wiki/Internet_telephony" title="Internet telephony" class="mw-redirect">Internet telephony</a> for example, while audio compression is used for CD ripping and is decoded by audio players.</p>
<h3><span class="editsection">[<a href="/w/index.php?title=Data_compression&amp;action=edit&amp;section=4" title="Edit section: Lossless">edit</a>]</span> <span class="mw-headline" id="Lossless">Lossless</span></h3>
<p>The <a href="/wiki/Lempel-Ziv" title="Lempel-Ziv" class="mw-redirect">Lempel-Ziv</a> (LZ) compression methods are among the most popular algorithms for lossless storage. <a href="/wiki/DEFLATE_(algorithm)" title="DEFLATE (algorithm)" class="mw-redirect">DEFLATE</a> is a variation on LZ which is optimized for decompression speed and compression ratio, therefore compression can be slow. DEFLATE is used in <a href="/wiki/PKZIP" title="PKZIP">PKZIP</a>, <a href="/wiki/Gzip" title="Gzip">gzip</a> and <a href="/wiki/Portable_Network_Graphics" title="Portable Network Graphics">PNG</a>. <a href="/wiki/LZW" title="LZW" class="mw-redirect">LZW</a> (Lempel-Ziv-Welch) is used in GIF images. Also noteworthy are the LZR (LZ-Renau) methods, which serve as the basis of the Zip method. LZ methods utilize a table-based compression model where table entries are substituted for repeated strings of data. For most LZ methods, this table is generated dynamically from earlier data in the input. The table itself is often <a href="/wiki/Huffman_coding" title="Huffman coding">Huffman encoded</a> (e.g. SHRI, LZX). A current LZ-based coding scheme that performs well is <a href="/wiki/LZX_(algorithm)" title="LZX (algorithm)">LZX</a>, used in Microsoft's <a href="/wiki/Cabinet_(file_format)" title="Cabinet (file format)">CAB</a> format.</p>
<p>The very best compressors use probabilistic models, in which predictions are coupled to an algorithm called <a href="/wiki/Arithmetic_coding" title="Arithmetic coding">arithmetic coding</a>. Arithmetic coding, invented by <a href="/wiki/Jorma_Rissanen" title="Jorma Rissanen">Jorma Rissanen</a>, and turned into a practical method by Witten, Neal, and Cleary, achieves superior compression to the better-known Huffman algorithm, and lends itself especially well to adaptive data compression tasks where the predictions are strongly context-dependent. Arithmetic coding is used in the bilevel image-compression standard <a href="/wiki/JBIG" title="JBIG">JBIG</a>, and the document-compression standard <a href="/wiki/DjVu" title="DjVu">DjVu</a>. The text <i>entry</i> system, <a href="/wiki/Dasher" title="Dasher">Dasher</a>, is an inverse-arithmetic-coder.</p>
<h2><span class="editsection">[<a href="/w/index.php?title=Data_compression&amp;action=edit&amp;section=5" title="Edit section: Theory">edit</a>]</span> <span class="mw-headline" id="Theory">Theory</span></h2>
<p>The theoretical background of compression is provided by <a href="/wiki/Information_theory" title="Information theory">information theory</a> (which is closely related to <a href="/wiki/Algorithmic_information_theory" title="Algorithmic information theory">algorithmic information theory</a>) for lossless compression, and by <a href="/wiki/Rate%E2%80%93distortion_theory" title="Rate–distortion theory">rate–distortion theory</a> for lossy compression. These fields of study were essentially created by <a href="/wiki/Claude_Shannon" title="Claude Shannon">Claude Shannon</a>, who published fundamental papers on the topic in the late 1940s and early 1950s. <a href="/wiki/Cryptography" title="Cryptography">Cryptography</a> and <a href="/wiki/Coding_theory" title="Coding theory">coding theory</a> are also closely related. The idea of data compression is deeply connected with <a href="/wiki/Statistical_inference" title="Statistical inference">statistical inference</a>.</p>
<p>Many lossless data compression systems can be viewed in terms of a four-stage model. Lossy data compression systems typically include even more stages, including, for example, prediction, frequency transformation, and quantization.</p>
<h3><span class="editsection">[<a href="/w/index.php?title=Data_compression&amp;action=edit&amp;section=6" title="Edit section: Machine learning">edit</a>]</span> <span class="mw-headline" id="Machine_learning">Machine learning</span></h3>
<div class="rellink boilerplate seealso">See also: <a href="/wiki/Machine_learning" title="Machine learning">Machine learning</a></div>
<p>There is a close connection between <a href="/wiki/Machine_learning" title="Machine learning">machine learning</a> and compression: a system that predicts the posterior probabilities of a sequence given its entire history can be used for optimal data compression (by using arithmetic coding on the output distribution), while an optimal compressor can be used for prediction (by finding the symbol that compresses best, given the previous history). This equivalence has been used as justification for data compression as a benchmark for "general intelligence" <sup id="cite_ref-0" class="reference"><a href="#cite_note-0"><span>[</span>1<span>]</span></a></sup>.</p>
<h3><span class="editsection">[<a href="/w/index.php?title=Data_compression&amp;action=edit&amp;section=7" title="Edit section: Data differencing">edit</a>]</span> <span class="mw-headline" id="Data_differencing">Data differencing</span></h3>
<div class="rellink relarticle mainarticle">Main article: <a href="/wiki/Data_differencing" title="Data differencing">Data differencing</a></div>
<p>Data compression can be seen as a special case of <a href="/wiki/Data_differencing" title="Data differencing">data differencing</a><sup id="cite_ref-1" class="reference"><a href="#cite_note-1"><span>[</span>2<span>]</span></a></sup><sup id="cite_ref-2" class="reference"><a href="#cite_note-2"><span>[</span>3<span>]</span></a></sup> – data differencing consists of producing a <i>difference</i> given a <i>source</i> and a <i>target</i>, with patching producing a <i>target</i> given a <i>source</i> and a <i>difference,</i> while data compression consists of producing a compressed file given a target, and decompression consists of producing a target given only a compressed file. Thus, one can consider data compression as data differencing with empty source data, the compressed file corresponding to a "difference from nothing". This is the same as considering absolute <a href="/wiki/Entropy_(information_theory)" title="Entropy (information theory)">entropy</a> (corresponding to data compression) as a special case of <a href="/wiki/Relative_entropy" title="Relative entropy" class="mw-redirect">relative entropy</a> (corresponding to data differencing) with no initial data.</p>
<p>When one wishes to emphasize the connection, one may use the term <b><a href="/wiki/Differential_compression" title="Differential compression" class="mw-redirect">differential compression</a></b> to refer to data differencing.</p>

<h4><span class="editsection">[<a href="/w/index.php?title=Data_compression&amp;action=edit&amp;section=13" title="Edit section: Example implementations">edit</a>]</span> <span class="mw-headline" id="Example_implementations">Example implementations</span></h4>
<ul>
<li><a href="/wiki/DEFLATE_(algorithm)" title="DEFLATE (algorithm)" class="mw-redirect">DEFLATE</a> (a combination of LZ77 and Huffman coding) – used by <a href="/wiki/ZIP_file_format" title="ZIP file format" class="mw-redirect">ZIP</a>, <a href="/wiki/Gzip" title="Gzip">gzip</a> and <a href="/wiki/Portable_Network_Graphics" title="Portable Network Graphics">PNG</a> files</li>
<li><a href="/wiki/LZMA" title="LZMA" class="mw-redirect">LZMA</a> used by <a href="/wiki/7-Zip" title="7-Zip">7-Zip</a></li>
<li><a href="/wiki/LZO" title="LZO" class="mw-redirect">LZO</a> (very fast LZ variation, speed oriented)</li>
<li><a href="/wiki/LZX_(algorithm)" title="LZX (algorithm)">LZX</a> (an LZ77 family compression algorithm)</li>
<li><a href="/wiki/Unix" title="Unix">Unix</a> <i><a href="/wiki/Compress" title="Compress">compress</a></i> utility (the .Z file format), and <a href="/wiki/Graphics_Interchange_Format" title="Graphics Interchange Format">GIF</a> use <a href="/wiki/LZW" title="LZW" class="mw-redirect">LZW</a></li>
<li>Unix <i><a href="/w/index.php?title=Pack_(program)&amp;action=edit&amp;redlink=1" class="new" title="Pack (program) (page does not exist)">pack</a></i> utility (the .z file format) used <a href="/wiki/Huffman_coding" title="Huffman coding">Huffman coding</a></li>
<li><a href="/wiki/Bzip2" title="Bzip2">bzip2</a> (a combination of the Burrows-Wheeler transform and Huffman coding)</li>
<li><a href="/wiki/PAQ" title="PAQ">PAQ</a> (very high compression based on <a href="/wiki/Context_mixing" title="Context mixing">context mixing</a>, but extremely slow; competing in the top of the highest compression competitions)</li>
</ul>
<ul>
<li><a href="/wiki/JPEG" title="JPEG">JPEG</a> (image compression using a discrete cosine transform, then quantization, then Huffman coding)</li>
<li><a href="/wiki/MPEG" title="MPEG" class="mw-redirect">MPEG</a> (audio and video compression standards family in wide use, using <a href="/wiki/Discrete_cosine_transform" title="Discrete cosine transform">DCT</a> and motion-compensated prediction for video)
<ul>
<li><a href="/wiki/MP3" title="MP3">MP3</a> (a part of the <a href="/wiki/MPEG-1" title="MPEG-1">MPEG-1</a> standard for sound and music compression, using subbanding and <a href="/wiki/Modified_discrete_cosine_transform" title="Modified discrete cosine transform">MDCT</a>, perceptual modeling, quantization, and Huffman coding)</li>
<li><a href="/wiki/Advanced_Audio_Coding" title="Advanced Audio Coding">AAC</a> (part of the <a href="/wiki/MPEG-2" title="MPEG-2">MPEG-2</a> and <a href="/wiki/MPEG-4" title="MPEG-4">MPEG-4</a> audio coding specifications, using <a href="/wiki/Modified_discrete_cosine_transform" title="Modified discrete cosine transform">MDCT</a>, perceptual modeling, quantization, and Huffman coding)</li>
</ul>
</li>
<li><a href="/wiki/Vorbis" title="Vorbis">Vorbis</a> (DCT based AAC-alike audio codec, designed with a focus on avoiding patent encumbrance)</li>
<li><a href="/wiki/JPEG_2000" title="JPEG 2000">JPEG 2000</a> (image compression using wavelets, then quantization, then entropy coding)</li>
<li><a href="/wiki/TTA_(codec)" title="TTA (codec)">TTA</a> (uses <a href="/wiki/Linear_predictive_coding" title="Linear predictive coding">linear predictive coding</a> for lossless audio compression)</li>
<li><a href="/wiki/Free_Lossless_Audio_Codec" title="Free Lossless Audio Codec">FLAC</a> (<a href="/wiki/Linear_predictive_coding" title="Linear predictive coding">linear predictive coding</a> for lossless audio compression)</li>
</ul>
